# 0025 前端统一错误处理

## 当前已有能力

- 后端业务异常已经返回统一 JSON，包含 `success`、`code`、`status`、`message`、`path`、`timestamp`、`fieldErrors`。
- 管理台已支持管理员登录、税率配置、大宗令牌发放、玩家资产查询、交易记录查询。
- 前端已使用 Pinia 管理管理台状态，并通过 Vite 构建。

## 本次完成内容

- 新增前端统一错误模块 `frontend/src/apiError.js`。
- 将后端错误码映射为前端用户提示，避免前端解析中文 `message`。
- 为错误对象保留 `code`、`status`、`path`、`fieldErrors`、建议操作 `action` 等结构化信息。
- 管理台 store 的接口请求统一使用结构化错误对象。
- 当接口返回 `AUTH_REQUIRED`、`AUTH_INVALID`、`AUTH_EXPIRED` 时，前端会自动清理本地管理员 token，避免继续使用失效登录态。
- 页面错误提示区新增：
  - 错误提示文案
  - 后端错误码
  - HTTP 状态与请求路径
  - 建议处理动作
  - 字段级校验错误列表
- 管理台关键按钮在请求处理中会禁用，降低重复提交风险。

## 验证结果

- 已执行：`npm.cmd run build`
- 结果：通过。
- 构建提示：ECharts 相关打包体积超过 Vite 默认 500 kB 警告线，属于当前管理台图表依赖带来的体积警告，不影响本次功能可用性。

## 风险与说明

- 当前统一错误处理已接入管理台，玩家侧页面尚未建设，因此暂未接入玩家侧 UI。
- 错误码提示文案目前维护在前端 `apiError.js`，后续可以根据产品语气继续细化。
- 大包体积警告后续可通过 ECharts 按需引入或动态加载交易图表解决。

## 下次建议目标

- 建设玩家端网页 MVP：登录/注册、资金与栏位概览、交易密码设置、商店购买、种植养殖、收获、交易站买卖。
- 玩家端 API 调用直接复用本次新增的统一错误处理，重点处理交易密码未设置、余额不足、库存不足、大宗令牌缺失。
- 后续再考虑将错误码对照表沉淀为前后端共享文档或 OpenAPI 描述。
