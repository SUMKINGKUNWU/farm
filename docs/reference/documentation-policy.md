# 文档与版本入库规则

## 目标

项目不会固定在单一开发环境中推进，因此每一阶段的需求、方案、实现状态和下一步计划都必须写入本地文档，保证任何环境接手时都能恢复上下文。

## 不可变版本文档规则

从当前阶段开始，版本记录采用只增不改策略：

- 每完成一段需求、设计或开发，都新增一份文档。
- 不修改历史版本文档。
- 如果历史方案需要修正，在新文档中说明“修正了什么、为什么修正、影响哪些模块”。
- 每份版本文档必须说明当前已有内容、本次新增内容、下次待做内容。

## 文件命名

版本文档建议命名为：

```text
docs/versions/0001-initial-product-and-technical-plan.md
docs/versions/0002-xxx.md
docs/versions/0003-xxx.md
```

编号递增，文件名使用英文小写和短横线，便于跨系统兼容。

## 每份版本文档模板

```markdown
# 版本编号 - 标题

## 当前已有

说明进入本阶段前，项目已经具备什么。

## 本次完成

说明本阶段新增或明确的内容。

## 产品范围

说明本阶段涉及的玩法、页面、后台、规则。

## 技术范围

说明本阶段涉及的前端、后端、数据库、缓存、接口或部署内容。

## 风险与漏洞

说明已识别的风险、漏洞、并发问题、经济系统问题。

## 下次待做

说明下一阶段建议推进的事项。
```

## 开发完成记录要求

后续每次完成开发时，版本文档需要至少包含：

- 新增或修改了哪些模块。
- 当前可运行功能是什么。
- 已执行的验证命令或测试结果。
- 未验证或存在风险的地方。
- 下一次开发建议从哪里开始。
